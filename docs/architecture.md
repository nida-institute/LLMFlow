# LLMFlow Architecture

## 1. Purpose
LLMFlow provides a declarative YAML-driven execution engine for LLM-assisted content pipelines. It standardizes prompt contracts, variable resolution, iteration, and output persistence while enabling domain‑specific extensions and multi‑repository content management.

## 2. Core Goals
- Deterministic, inspectable pipelines (lint + contract validation)
- Provider/model agnostic (via `llm` package + plugins)
- Human‑in‑the‑loop editing after generation (Git-managed outputs)
- Separation of engine (public repo) and resource collections (per‑type repos)
- Scalable to large content graphs (scene lists, lexicons, guides)
- Future integration with MCP (Model Context Protocol) for dynamic context and tool access

## 3. High-Level Component Map
- CLI Layer (`cli.py`): argument parsing, dispatch (`run`, `lint`, `list`, version)
- Runner (`runner.py`): pipeline orchestration (step dispatch, context evolution)
- Variable Resolution (`resolve()`): `${var}`, nested attributes, list indexing
- Prompt Rendering (`render_prompt()`): load `.gpt` file + single‑brace substitution
- Template Rendering (`render_markdown_template()`): `{{var}}` + `${var}` substitution for markdown assembly
- LLM Adapter (`utils/llm_runner.py`): validates config, calls `llm` package, optional JSON parsing
- Linter (`utils/linter.py`): schema + prompt contract enforcement, path resolution
- Plugins (`plugins/loader.py`, `plugins/contrib/*.py`): dynamic capability injection (e.g., XPath extraction)
- Domain Utilities (`utils/data.py`): reference parsing (Bible, lexicon mapping, etc.)
- Output Persistence (`saveas`): filesystem writes (auto directory creation)
- Future MCP Layer: standardized context/tool exposure for interactive editing and pipeline augmentation

## 4. Data Flow (Simplified)
1. Load pipeline YAML → validate schema + prompt contracts
2. Initialize context with `variables:` + CLI overrides
3. For each step:
   - Resolve inputs (`${...}`) into concrete values
   - LLM step:
     - Render prompt file
     - Call model via `llm`
     - Parse JSON if `output_type: json`
   - Function step:
     - Invoke Python function with resolved arguments
   - For-each step:
     - Iterate list; isolate per‑item context; collect via `append_to`
4. Persist outputs (`saveas`) + extend context
5. Post-run: outputs committed in resource-specific repos (human edits)

## 5. Prompt Contract Model
`.gpt` file header:
```
<!--
prompt:
  requires:
    - passage
    - scene
  optional: []
  format: Markdown
  description: ...
-->
```
Linter checks: presence of all `requires` in `prompt.inputs`. Substitution currently supports `{key}` (single brace) replacement; prompts use `{{var}}` but substitution path will be normalized (see Planned Refactor).

## 6. Templates
Markdown templates use `{{var}}` (direct dict substitution) and `${var}` (context resolution). Multi-pass approach enables late binding from evolving context (e.g., aggregated scene lists).

## 7. Output Strategy (Multi-Repo)
- Core engine repo: runtime + shared utilities (public).
- Resource repos (e.g., leaders’ guides, lexicons, exegetical notes):
  - Hold pipelines specific to that content domain
  - Generated artifacts stored under `outputs/`
  - Human editors refine content in Git (commit status metadata)
- Cross-repo interoperability via shared installation of `llmflow` (editable or versioned release).

## 8. Obsidian Vault Integration
- Resource repos can be structured as Obsidian vaults:
  - Each generated file becomes a note
  - Inter-note links inserted post-generation (future function step: link resolver)
  - Status front-matter (e.g., `status: needs-review`, `quality: draft`) manually maintained and versioned
  - Index notes auto-generated by pipeline functions (lexicon indices, scene catalogs)

## 9. MCP Integration (Planned)
Goals:
- Expose pipeline context as MCP resources
- Register tools for:
  - Regenerating individual steps
  - Diffing model output vs edited human version
  - Contract introspection (list required prompt inputs)
- Live status synchronization: MCP client queries vault metadata (e.g., readiness state)
Implementation Plan (phased):
1. MCP Adapter Module (`mcp/adapter.py`) wrapping runner APIs
2. Resource Provider: surfaces pipeline variables + outputs as structured resources
3. Tool Provider: `rerun_step`, `validate_contract`, `extract_links`
4. Context Sync: map Obsidian front‑matter → pipeline feedback loop

## 10. Editing Workflow
1. Generate initial content via pipeline.
2. Human edits in resource repo / Obsidian.
3. Optional reconciliation step (future): detect divergence from model prompts → mark outdated sections.
4. Use MCP tools (future) to selectively regenerate / align content.

## 11. Error Handling Philosophy
- Hard validation failures: stop before execution (schema / contract)
- Step failures: logged; current behavior continues unless critical (future configurable)
- JSON parse failures: return raw text if parsing cannot succeed (future: strict flag)

## 12. Performance & Scaling
- Iterative `for-each` loops handle scene arrays / lexicon entries
- Append accumulation supports large batch construction
- Future optimization: caching layer keyed by (model, prompt hash, inputs)

## 13. Security / Compliance
- No embedded secrets; relies on `llm` package key management
- Output repositories remain separate → easier review of generated content for licensing / attribution
- Apache 2.0 license with institutional copyright (Biblica, Inc.)

## 14. Planned Refactors
| Area | Change | Rationale |
|------|--------|-----------|
| Prompt substitution | Unify on `{{var}}` | Consistency with templates |
| Inputs dict usage | Merge explicitly before replacement | Clarity + contract leverage |
| MCP layer | Add adapter | Interactive tooling |
| Link/index builder | Dedicated function module | Obsidian cross-reference automation |
| Caching | Hash-based response cache | Reduce cost & latency |

## 15. Extension Patterns
- New function step: add to `utils/*` or domain package, reference via `function: your.module.fn`
- New plugin: register via `plugins/loader.py` using `plugin_registry.register("name", callable)`
- MCP tool: expose standardized signature `(context, params) -> result`

## 16. Deployment / Usage
- Install `llmflow` once (editable or released)
- Clone resource repo → run pipeline → commit edited outputs
- CI (future): lint pipelines + validate prompt contract consistency

## 17. Constraints
- Current prompt substitution mismatch (`{{var}}` vs `{var}`) → document + refactor
- No native concurrency yet
- Assumes filesystem write access

## 18. Summary
LLMFlow isolates the orchestration layer from domain artifacts, enabling reproducible LLM-assisted content generation with human editorial control, multi-repo distribution, planned MCP integration, and vault-aware metadata workflows.
